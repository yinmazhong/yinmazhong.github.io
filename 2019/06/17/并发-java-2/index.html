Java 线程

生命周期

  1. 初始状态：线程已经在被创建，编程语言级别的被创建，非操作系统级别的，因为还不允许被分配cpu。
  2. 可运行状态：可以被分配cpu
  3. 运行态：正拥有某cpu
  4. 休眠态：阻塞，放弃cpu使用权
  5. 终止态：执行完毕，或者出现异常进入终止态

Java　中将可运行态与运行态合并了，应为jvm无法调度cpu,
但细化了休眠状态

分别是：

  1. NEW（初始化状态）
  2. RUNNABLE（可运行 / 运行状态）
  3. BLOCKED（阻塞状态）
  4. WAITING（无时限等待）
  5. TIMED_WAITING（有时限等待）
  6. TERMINATED（终止状态）

虽然阻塞态被细分成了仨，，但是从操作系统角度考虑，只要java线程处于这三种状态之一，那么这个线程就永远没有cpu使用权

为什么要使用多线程？

解：　提高性能

如何度量性能？延迟和吞吐量

延迟指的是发出请求到收到响应这个过程的时间；延迟越短，意味着程序执行得越快，性能也越好。

吞吐量指的是单位时间内能处理请求的数量

分别是时间维度和空间维度

怎么办？
降低延迟，提高吞吐量

  1. 优化算法
  2. 将硬件的性能发挥到极致

在并发编程领域，提升性能本质上就是提升硬件（I/O和CPU）的利用率

创建多少线程合适？
  • 对于 CPU 密集型的计算场景，理论上“线程的数量 =CPU 核数”就是最合适的。不过在工程上，线程的数量一般会设置为“CPU 核数 +1”，这样的话，当线程因为偶尔的内存页失效或其他原因导致阻塞时，这个额外的线程可以顶上，从而保证 CPU 的利用率。
  • 对于 I/O 密集型计算场景，最佳的线程数是与程序中 CPU 计算和 I/O 操作的耗时比相关的，我们可以总结出这样一个公式：最佳线程数 =CPU 核数 * [ 1 +（I/O 耗时 / CPU 耗时）]

为什么局部变量是线程安全的？

局部变量在方法内，局部方法调用是通过栈完成的，栈是线程私有的，所以局部变量是安全的

面向对象的发编程模型(我觉得大多数情况下没有面向函数来的好)

封装共享变量

将共享变量作为对象属性封装在内部，对所有公共方法制定并发访问策略

不会改变的尽量加final关键字修饰

识别共享变量间的约束条件

约束条件决定了并发访问策略：库存，有上限和下限
下限要小于上限

制定并发访问策略

1.避免共享
